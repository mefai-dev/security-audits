## Executive Summary

Brigid Forge is a BNB Smart Chain token-launch and liquidity-lock platform ("launch your token, lock liquidity, enable real-time monitoring"). Unlike many audited projects, this codebase is **original work by a single developer**, not a fork, and the public development log shows a genuinely security-conscious process (Slither runs, adversarial test orchestration, prior hardening passes). **No independent third-party audit exists for this project**, despite the platform itself recommending audits to the projects that launch on it. This report is, to our knowledge, the first real security review Brigid Forge has received.

**The headline finding is a real, code-level gap in an otherwise well-intentioned safety design, not a marketing exaggeration:**

**BrigidLaunchOrchestrator's "Custom Launch" path does not enforce that the announced liquidity reserve is actually committed to the pool.** `activateLaunch()` only checks that the liquidity a deployer commits does not *exceed* the reserve they announced (`tokenAmountDesired > launch.lpReserve` reverts); there is currently **no lower bound**. Because the trading pair is brand new at that point, PancakeRouter's first-mint accepts whatever amount is offered with no partial-fill penalty, and the **unused remainder of the announced reserve is transferred back to the deployer's own wallet as ordinary, unlocked, freely tradeable tokens**, at the same moment the launch flips to `CERTIFIED` and the UI/Beacon system announces a locked reserve. In principle, a deployer could advertise "40% of supply locked in liquidity," lock a real position covering only a small fraction of that promise, and retain most of the rest as liquid tokens. Two independent reviews concluded this path is technically exploitable today; it requires no external cooperation, and would not be visible to a user relying only on the on-chain "LP locked" proof. We found no evidence that this has actually been used this way; the concern is that the code does not currently prevent it.

## 1. Missing Minimum-Commitment Floor in Custom Launch Liquidity (Unbounded Reserve Draw)

**Severity: HIGH** (assessed as technically exploitable by 2 independent adversarial reviews; no evidence found that it has been exploited in practice)

**CLAIM (site + on-chain "Certified" status):**
> "Lock liquidity." "4-stage verification: vault deployment, LP lock, ownership renouncement, monitoring activation." A `CERTIFIED` launch implies the announced reserve was committed to the pool and locked.

**REALITY (code-traced, `contracts/imports/BrigidLaunchOrchestrator.sol`):**
- `activateLaunch()` (lines 375-377): `if (params.tokenAmountDesired > launch.lpReserve) revert InsufficientEscrowedReserve(...);` enforces an upper bound only, no floor.
- Line 370: `if (msg.value == 0) revert ZeroAmount();` means any tiny positive value passes.
- Lines 417-421: `uint256 reserveRemaining = launch.lpReserve - amountToken; launch.lpReserve = 0; if (reserveRemaining > 0) { tokenContract.safeTransfer(msg.sender, reserveRemaining); }`. The unused reserve goes directly to the deployer, as plain transferable tokens with no vesting and no lock.
- Contrast: the sibling **Meme Launch** path (`BrigidMemeLaunchOrchestrator.sol`) structurally commits at least 75% of supply to the pool (`devAllocationBps` capped at 2500, i.e. 25%). The **Custom Launch** path has no equivalent floor at all.
- `BrigidLaunchRegistry.sol` explicitly documents (comments, lines 19-36) that it does **not** track LP pair/lock state; nothing else in the system enforces the missing invariant.

**EXPLOIT SCENARIO:**
1. Deployer calls `createLaunch` with `tokenSupply = 1,000,000,000e18` and `lpReserveRaw = 400,000,000e18` (marketed as "40% locked liquidity"); the rest is allocated to vaults.
2. Community sees `LaunchCreated`/`getLaunch()` showing `lpReserve = 400,000,000e18` and trusts it.
3. Deployer calls `activateLaunch` with `tokenAmountDesired = 1,000,000e18` (0.25% of the reserve, safely above PancakeSwap V2's `MINIMUM_LIQUIDITY` floor) and `msg.value = 0.0025 BNB`.
4. Because it's the pair's first mint, the router consumes exactly those tiny amounts. `reserveRemaining ≈ 399,000,000e18` is sent straight to the deployer's wallet.
5. `BrigidManagedLPLock` genuinely locks a real LP position for at least 180 days, but that position backs only 0.25% of the promised reserve. The launch is `CERTIFIED`, Beacon announces "liquidity locked," and the deployer holds about 40% of supply, fully liquid, sellable into the near-empty pool before or as retail buyers arrive.

**IMPACT:** This meaningfully weakens the assurance the Custom Launch path is meant to provide. A deployer acting in bad faith, or simply misunderstanding the mechanics, could point to genuine, verifiable on-chain "locked LP" proof while retaining most of the announced float as liquid balance. We want to be clear this is a **gap in the code's current guarantees**, not a claim that any specific launch has misused it. Closing it would bring the Custom Launch path's real-world behavior in line with what it already advertises.

**Reassuring context (on-chain, checked directly):** as of this report, `totalLaunches()` on the Orchestrator is **1**; the only launch that exists is Brigid's own BRIGID token. Reconstructing that launch from current on-chain state (pool reserves, LP token supply, deployer's current balance) is consistent with the **full** announced reserve having been committed to the pool, not a partial draw. The deployer's current BRIGID balance is 0, and the LP supply locked in `BrigidManagedLPLock` matches the pair's total minted LP minus only the standard 1000-wei `MINIMUM_LIQUIDITY` burn, meaning effectively 100% of the minted LP is locked. We could not replay the exact `LaunchActivated` event directly since public RPC nodes rejected the log range, so this is an arithmetic reconstruction rather than a transaction replay, but it is a positive data point that the one real-world case we could check does not show the gap having been used.

**RECOMMENDATION:** Enforce a minimum-commitment floor in `activateLaunch()`, for example requiring `tokenAmountDesired >= X% of launch.lpReserve` (mirroring the Meme Launch path's `devAllocationBps` cap), or require the full reserve to be posted atomically at launch creation rather than allowing a deployer-chosen partial draw at activation time.

## 2. `feeQuoteSigner` Has Unbounded Blast Radius (No Price Floor/Ceiling)

**Severity: MEDIUM** (confirmed exploitable by 2 independent adversarial reviews)

**REALITY:** `_verifyFeeQuote()` (`BrigidLaunchOrchestrator.sol:651-665`) validates payer, tier, expiry, nonce-replay, and the ECDSA signature of a `FeeQuote`, but **never checks `quote.feeAmount` against the tier's actual configured price** (`launchTierFees[tier]`, checked via `_launchFeeForTier`). The signed `feeAmount` is an arbitrary field chosen by whoever holds the `feeQuoteSigner` key.

**EXPLOIT SCENARIO:** If the `feeQuoteSigner` hot key (plausibly a backend/API key, separate from the cold `owner` key) is leaked, phished, or misused, the holder can sign themselves a quote with `feeAmount = 1 wei` for the most expensive tier and call `createLaunchWithFeeQuote`, bypassing the intended fee structure entirely, silently, indefinitely, until the key is rotated. The rest of the signature machinery (domain separator, replay protection, expiry) is implemented correctly; only the price-floor invariant is missing.

**IMPACT:** Full, silent revenue-bypass on compromise of a single operational key that is more exposed, by design meant to be usable by a live backend, than the cold owner key.

## 3. Unbounded On-Chain Loops in Staking (DoS on Surplus Recovery + Unbounded Vesting Tail)

**Severity: MEDIUM** (confirmed exploitable by 2 independent adversarial reviews for both the loop and the vesting tail)

**REALITY:**
- `BrigidStaking.totalClaimableRewards()` unconditionally loops over **every lot ID ever minted** (`for (uint256 lotId = 1; lotId < nextLotId; ++lotId)`), with no paginated/bounded variant wired into the surplus-recovery path. A bounded variant (`totalClaimableRewardsRange(startLotId, count)`, lines 312-330) already exists elsewhere in the same contract; the codebase clearly knows this pattern, since `claimRange`/`claimableRange` (lines 162-172, 246-257) already use an equivalent bounded loop via the shared `_boundedEnd` helper (lines 419-427), but it is not used by `totalRequiredBacking()`/`surplusTokenBalance()`/`recoverSurplusStakingTokens()`, which only ever call the unbounded `totalClaimableRewards()`. This is the sole path feeding `surplusTokenBalance()` into `recoverSurplusStakingTokens()`. As the platform is used over time and `nextLotId` grows, this function's gas cost grows without bound and will eventually exceed the block gas limit, **permanently disabling the owner's ability to recover surplus tokens**. This is not a fund-loss bug, but a genuine, self-inflicted denial-of-service on legitimate platform operations.
- Separately, `BrigidVaultFactory`'s bounds check caps `MAX_START_OFFSET` (5 years) and `MAX_CLIFF_DURATION` (10 years), but `intervalDuration`/`intervalCount` (which determine the length of the vesting *tail* after the cliff) are **completely unbounded**, so the two documented caps do not actually bound the total vesting horizon a vault can be configured with.

**IMPACT:** Low fund-risk since there is no theft path, but a real availability risk that grows worse the longer and more successfully the platform is used, an unusual "success causes future failure" pattern worth fixing before scale.

## 4. No Anti-Bot/Anti-Snipe Protection on Launch Tokens

**Severity: MEDIUM** (confirmed exploitable by 2 independent adversarial reviews)

**REALITY:** Neither `BrigidLaunchToken.sol` nor `BrigidStandardMemeToken` (inside `BrigidMemeLaunchOrchestrator.sol`) implements any max-transaction limit, per-block buy cap, or trading-enabled gate. `activateLaunch()`/`launchMeme()` are ordinary public mempool transactions, so the resulting pool price is fully exposed to MEV sandwich/sniper bots the instant the transaction is visible, before genuine retail buyers can react.

**IMPACT:** This is a systemic risk shared by most AMM-based launches generally, but is worth flagging explicitly because the platform markets itself on launch-day safety. A sniper capturing the first block's supply at launch price directly undermines that promise, independent of Finding #1.

## 5. "Security Audit" Language Could Be Clarified (No Independent Firm Identified)

**Severity: MEDIUM** (a terminology and clarity gap rather than a factual misstatement)

**CLAIM (public development log, 2026-05-31 entry):**
> "Remediated the 2026-05-31 security audit findings... Recorded residual-risk signoff for the Slither pass after cleanup, keeping the public audit posture aligned with the current release candidate."

**REALITY:** We could not find an independent audit firm name, engagement report, or third-party portfolio entry anywhere in the public repository, website, or a targeted web search (a search for `BrigidForge audit "security audit" "third party audit" smart contract` returned **0 results**). The hardening *patterns* described in the log genuinely exist in the code (dual balance-delta checks, nonce/overflow protections, and similar), so the underlying work looks real. What we could not confirm is whether "audit" here refers to a paid, independent, named review, or to an in-house Slither pass with the developer's own sign-off; the log itself does not specify. The site (`/brigid-launch-standard/`) already discloses elsewhere that "Brigid Launchpad verification is not the same as a smart contract audit," a good practice that could simply be extended to this specific line in the dev log.

**IMPACT:** A reader could reasonably read "security audit" as implying an independent third party was involved. We think this is most likely an unintentional wording gap rather than a deliberate claim, given how transparent the rest of the public dev log is, but it is worth the team clarifying explicitly (for example naming the tool/process, or adding "internal" before "audit") so the language matches the reality precisely.

## 6. Governance Centralization: Two Separate Single-EOA Owners, No Multisig/Timelock

**Severity: MEDIUM** (on-chain confirmed)

**REALITY (on-chain, direct `eth_call`, 2026-07-02):**
- `BrigidLaunchOrchestrator` (`0x394408...0e809`) `owner()` = `0x45a1f4fa...b11e`.
- `BrigidVaultFactory` (`0x226065...382dfb`) `owner()` = `0x853453...ea22fe`, a different address. Note that `feeRecipient()` is not a function on VaultFactory itself; this same address separately matches Orchestrator's own `feeRecipient` field.
- Neither resolves to a Gnosis Safe or any other multisig. The Factory owner (`0x853453...ea22fe`) is a plain EOA with no bytecode. The Orchestrator owner (`0x45a1f4fa...b11e`) carries a small EIP-7702 delegation (23 bytes, pointing to a smart-account implementation), a "smart EOA" in the MetaMask/Ambire sense, still controlled by a single private key with no multi-signer threshold. The "not a multisig" conclusion holds, but it is not literally a bare, codeless EOA.
- `BrigidRevenueRouter`'s owner can redirect the treasury/burn/staking destination addresses at will, immediately before triggering `distribute()`, giving discretionary control over the entire accumulated protocol revenue pool.
- `BrigidSubscription`'s owner can unilaterally grant an unpaid project a subscription or cancel a paying customer's subscription via `setProjectExpiry()`, bypassing the `purchase()` payment flow entirely.
- `BrigidMemeLaunchOrchestrator.rescueNative()/rescueToken()` give the owner real fund-withdrawal power, restricted only by a blocklist rather than an allowlist.

**IMPACT:** None of this is a hidden backdoor; it is disclosed, ordinary `Ownable` centralization, consistent with an early-stage, single-developer project. But it means every privileged action (fee changes, revenue routing, subscription grants, rescue functions) currently depends on the operational security of two individual private keys with no timelock and no second-signer requirement. Given the project's own token-utility claims include revenue-sharing, this centralization is directly relevant to token holders, not just launch customers.

**Open question (could not be resolved from public sources):** why do the Orchestrator and the Factory have *different* owner addresses? It could be a deliberate separation of concerns, for example a deploy key versus a treasury key, or simply an operational inconsistency. We could not determine this without non-public information, and are not speculating further.

## 7. Live Status of Staking/Revenue-Share Utility Could Not Be Confirmed

**Severity: MEDIUM** (flagged as an open question, not confirmed either way)

**CLAIM (`/token/` page, per site content review):** BRIGID token utility includes "staking, revenue-share (40%), token burn, ecosystem incentives."

**REALITY:** The public development log's only staking/revenue-router deployment entry (2026-05-02) explicitly describes a **BSC testnet** deployment ("Rotated the active BSC testnet staking and revenue stack..."). No mainnet address for `BrigidStaking`, `BrigidRevenueRouter`, or `BrigidSubscription` could be found anywhere in public site content, the GitHub repo, or via on-chain forensics from the known owner addresses (BscScan's contract-creation history could not be enumerated without an API key in this review). It is therefore **unclear whether the staking/revenue-share utility described on the token page is live on mainnet today, or is future/testnet-only functionality being described in present tense.**

**RECOMMENDATION:** Brigid Forge should publish the mainnet addresses, or explicitly label the feature "coming soon" or "testnet," for every contract referenced in the token's utility claims, exactly as it already does for the Vault/Factory/Orchestrator addresses.

## Conclusion

**Bottom line for users and prospective launch customers:** Brigid Forge is built on genuinely original code by a security-conscious single developer, with unusually transparent public documentation including a real, detailed dev log, a materially different profile from a typical fork-and-hide project. This report did find a **real gap in the platform's core safety promise**: the Custom Launch path does not currently enforce that an announced liquidity reserve is actually committed, which two independent reviews assessed as technically exploitable, and which would not be visible from the on-chain "LP locked" proof alone. Combined with the absence of any independent third-party audit, single-EOA (non-multisig) control over all deployed contracts, and open questions about which token-utility features are live on mainnet today, **we'd suggest treating "Certified" status as informative rather than as complete proof that the full announced liquidity was locked, until Finding #1 is addressed.** None of this changes our assessment that the underlying engineering approach is unusually careful for a project at this stage; these are fixable gaps, not evidence of bad intent.

## Recommendations

**For users / launch customers:**
1. Do not treat `CERTIFIED` status alone as proof that the full announced liquidity reserve was locked. Independently check the actual LP token balance locked in `BrigidManagedLPLock` against the originally announced `lpReserve`.
2. Confirm directly with the team whether staking/revenue-share utility is live on mainnet before valuing the token on that basis.
3. Understand that both the Orchestrator and Factory owner keys are single EOAs with no timelock; treat operational key security as equivalent to the entire platform's trust boundary.

**For the project (to close the gap and become genuinely audit-grade):**
1. **Fix Finding #1 immediately.** Enforce a minimum-commitment floor in `activateLaunch()` before allowing any launch to reach `CERTIFIED`.
2. Add an on-chain floor/ceiling check for `feeQuoteSigner`-signed fee amounts against the configured tier price (Finding #2).
3. Bound `intervalDuration`/`intervalCount`, and wire the bounded pattern the codebase already has (`totalClaimableRewardsRange`, or an equivalent) into `totalRequiredBacking()`/`surplusTokenBalance()`/`recoverSurplusStakingTokens()`. The primitive exists in the contract, it just isn't used on this path (Finding #3).
4. Commission a real, named, independent smart-contract audit and publish the report; do not describe an internal Slither pass as a "security audit" without qualification (Finding #5).
5. Migrate both owner keys to a multisig with a timelock, and publish mainnet addresses, or a clear "not yet live" status, for the staking/revenue stack.

## Methodology

1. Front-end and JS-rendered page-by-page reading of all public documentation pages (`/transparency/`, `/token/`, `/guides/`, `/brigid-launch-standard/`, `/disclosures-risk-notice/`, `/token-launch-technical-checklist/`, `/beacon/`) via a real-Chrome-fingerprint fetch pipeline.
2. Full clone and git-history forensics of the public contracts repository, checking commit authorship and originality.
3. Two independent rounds of line-by-line Solidity review across all 13 public contract files. Round 1 covered ownership/admin-function mapping and dev-log claim verification. Round 2 specifically targeted re-entrancy across all fund-moving functions, signature/nonce security, vesting/reward arithmetic, PancakeSwap integration risk, a full access-control matrix, and unbounded-loop DoS risk. Every finding was subjected to at least two independent adversarial verification passes.
4. Direct BNB Smart Chain JSON-RPC verification (`eth_call`/`eth_getCode`) for contract versus EOA determination, function-selector fingerprinting to identify unlabeled addresses, owner/admin role resolution, and PancakeSwap pair reserve measurement.
5. Web security header scan, DNS (SPF/DMARC) verification, and social-account activity check.
6. Targeted search for independent third-party audit evidence, combining a web search with a BscScan verification-status check.
7. A dedicated final verification pass that re-derived every claim in this report from primary sources (code, on-chain calls, live site) independent of the report's own text. This caught and corrected several factual details before publication, notably total supply, an address, and the BscScan verification status.

## Disclaimer

This report is based on public data available on 2026-07-01 and 2026-07-02 and is provided for informational and consumer-protection purposes. It is **not financial advice**. On-chain metrics can change.

This assessment was conducted **entirely from public sources and read-only on-chain calls, with no intrusion, exploitation, or unauthorized access of any kind, and no transaction was ever signed or broadcast.** Every finding is derived from the team's own publicly published source code, public documentation, and read-only BSC JSON-RPC calls. The centralization findings, contract identities, and liquidity depth are all confirmed via direct BSC RPC calls. Finding #1 is a code-level logical analysis of publicly available source code and was not executed against the live contract.

Originality of source code, transparent public documentation, and a security-conscious development process are not, by themselves, proof of safety. The conclusions here rest on the combination of a genuine but unaudited codebase, one concretely exploitable missing invariant in the core "locked liquidity" guarantee, unbounded-loop availability risks, single-EOA administrative control with no timelock, and an audit claim that could not be substantiated by any independent source.
